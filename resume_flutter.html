Section 02
실서비스에서 해결한 문제들
PROBLEM 01
셋톱박스 3대 페어링 상태 — Riverpod으로 연결 / 기기 / 콘텐츠 단위 분리
최대 3대 기기 상태 혼재 → 연결 / 기기 / 콘텐츠 단위 분리 → StateNotifierProvider 설계 → 상태 변화 즉시 반영
헬로tv → 오초이스 마이그레이션에서 셋톱박스 최대 3대 페어링 상태를 단일 Provider로 관리하면 상태 혼재·렌더링 오류 발생. 연결 상태 / 기기 노출 / 콘텐츠 접근 권한을 별도 StateNotifierProvider로 분리해 각 단위의 상태 변화가 독립적으로 즉시 반영되도록 설계. 이후 신규 기능 추가 시에도 영향 범위가 명확해져 유지보수 난이도 감소.
PROBLEM 02
쇼츠형 화면 오디오 이중 출력 — 플레이어 생명주기 분리로 해결
스크롤 시 이전 영상 오디오 잔존 → 원본 라이브러리 소스 레벨 분석 → 생성 / 재생 / 종료 생명주기 명시 분리 → 오디오 겹침 완전 해결
딜라이브 마이그레이션에서 쇼츠형 피드 스크롤 시 이전 영상의 오디오가 겹쳐 출력되는 문제 발생. 플레이어 라이브러리 API 문서만으로 해결 불가 — 원본 라이브러리 소스 코드를 직접 분석해 내부 생명주기 처리 방식을 파악. 영상 전환 시 플레이어 생성 / 재생 / 종료 단계를 명시적으로 분리 호출해 오디오 겹침 문제 완전 해결.
PROBLEM 03
NCG DRM → Multi DRM 전환 — 단일 플레이어로 통합
서비스마다 다른 DRM 방식 → 단일 플레이어 인터페이스 설계 → 라이선스 검증 흐름 통합 → 서비스 무관 동일 사용자 경험
딜라이브 앱(NCG DRM)과 헬로tv 앱(Multi DRM: Widevine/FairPlay)의 DRM 방식이 달라 동일 플레이어로 통합이 어려운 문제. DRM 방식을 추상화한 단일 플레이어 인터페이스를 설계해 내부에서 DRM 타입을 분기 처리. 사용자에게는 서비스별 기술 차이와 무관하게 동일한 시청 흐름 제공.
PROBLEM 04
Flutter 2 → 3 메이저 버전업 — 라이브러리 의존성 전체 재검토
Flutter 2 → 3 업그레이드 → Null Safety 전환 + 의존성 충돌 분석 → 호환 버전 재검토 → 안정적 업그레이드 완료
딜라이브 마이그레이션 프로젝트에서 Flutter 2 기반 코드를 Flutter 3으로 업그레이드. DRM 라이브러리, 미디어 플레이어, 상태관리 라이브러리 등 전체 의존성을 검토해 Flutter 3 호환 여부 확인 및 호환 버전으로 교체. Null Safety 마이그레이션이 필요한 코드를 직접 수정해 빌드 오류 없이 안정적으로 업그레이드 완료.
PROBLEM 05
WebView + JS Bridge — FCP 39% 단축하며 운영 유연성 확보
앱 배포 주기 의존 + 초기 로딩 지연 → 탐색 화면 WebView 분리 → JS Bridge 양방향 통신 설계 → FCP −39% · 즉시 반영 운영 구조
탐색 화면의 잦은 콘텐츠 변경이 앱 배포 주기에 의존하고, 초기 로딩 시간이 길어 이탈이 발생하는 문제. React/TypeScript로 탐색 화면을 분리하고 Flutter ↔ WebView JavaScript Bridge 양방향 통신 구조를 설계. Lazy Loading·스켈레톤 UI·CDN 적용으로 FCP 39% 단축, LCP 10.8% 개선. 앱 재배포 없이 콘텐츠·배너·이벤트를 즉시 반영 가능한 운영 구조 완성.
변경 전
탐색 화면 변경 → 앱 코드 수정 → 빌드 → 앱스토어 심사(1~3일) → 배포
변경 후
탐색 화면 변경 → React 웹 배포 → 즉시 반영 (앱 배포 없음)

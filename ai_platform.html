필수 항목 02
Failure & Growth

실패에서 배운 것

실제 운영 장애와 설계 실수에서 얻은 교훈입니다.

상황 원인 개선 교훈
운영 인프라 설계 실수 — Proxy 서버 IP 변경으로 대형 파트너사 연동이 반복 중단됐다
상황 (Situation)
홈초이스에서 LG헬로비전(가입자 343만) 파트너사 연동 API를 운영하던 중, 어느 날부터 외부 SO 서버의 방화벽이 우리 Proxy 서버를 차단하는 장애가 반복 발생했습니다. 매번 파트너사에 IP 화이트리스트 등록을 요청해야 했고, 일주일 이상 서비스가 중단되는 상황이 세 차례 이어졌습니다.
원인 (Root Cause)
원인은 AWS 로드밸런서 환경에서 EC2 인스턴스 재시작 시 Proxy 서버의 Public IP가 자동으로 변경된다는 점을 초기 설계 시 전혀 고려하지 않은 것이었습니다. 파트너사의 방화벽은 고정 IP만 허용하는데, 저는 IP가 변경될 수 있다는 인프라 특성을 숙지하지 못한 채 설계했습니다. 그 결과 배포·재시작마다 화이트리스트 재등록 절차가 필요한 구조가 반복되었습니다.
개선 방향 (Solution)
원인을 파악한 뒤 EC2 기반 Proxy 서버에 Elastic IP를 할당하여 고정 IP 구조로 재설계했습니다. 이후 파트너사 방화벽 등록을 단 한 번 진행했고, 이후 재시작·배포에도 IP가 변하지 않아 동일 장애가 완전히 사라졌습니다. 추가로 Proxy 서버의 SSL 인증서 미적용으로 통신 실패가 발생했던 문제도 발견해 전 구간 HTTPS 전환까지 단독으로 수행했습니다.
교훈 (Lesson)
이 경험 이후, 외부 시스템과 연동하는 서비스를 설계할 때는 반드시 인프라 환경 특성(IP 변동성, 네트워크 정책)을 먼저 리스트업합니다. 특히 AI API 서비스처럼 외부 LLM API와의 통신이 핵심인 시스템에서는, API Key 관리·IP 정책·Rate Limit 등 외부 의존성의 변동 조건을 설계 단계에서 명시적으로 다루는 습관이 생겼습니다.
🔧 Elastic IP 재설계 🔒 HTTPS 전환 완료 ✅ 동일 장애 재발 없음 📋 인프라 의존성 설계 체크리스트화
상황 원인 개선 교훈
Study Tracker API — ORM 기본 설정을 믿었다가 N+1 문제로 쿼리 1,000건 이상 발생
상황 (Situation)
Study Tracker API에서 사용자의 학습 목록을 조회할 때, 로컬에서는 문제없어 보였지만 데이터를 100건 이상 넣고 나서야 쿼리가 수백 건 발생하고 있음을 발견했습니다. API 응답 시간이 데이터 건수에 비례해 선형으로 증가하는 명백한 성능 문제였습니다.
원인 (Root Cause)
원인은 Spring Data JPA의 기본 Lazy Loading 전략을 그대로 사용해 N+1 문제가 발생한 것이었습니다. 설계 단계에서 "JPA가 알아서 최적화해줄 것"이라고 가정한 채 조회 쿼리를 설계했고, 실제 발생 쿼리 수를 측정하지 않은 것이 실수였습니다.
개선 방향 (Solution)
JPQL Fetch Join과 복합 인덱스 2개(user_id,id / user_id,completed,id)를 직접 설계해 N+1을 해결했습니다. 추가로 @Cacheable/@CacheEvict + Redis TTL 60초로 반복 조회를 캐싱해 DB 부하를 대폭 줄였습니다. 이후 P99 응답 시간이 의미 있게 개선된 것을 직접 측정하고 확인했습니다.
교훈 (Lesson)
이후 모든 데이터 조회 API를 설계할 때 "몇 건일 때 쿼리가 몇 개 발생하는가"를 먼저 측정하는 습관이 생겼습니다. AI 플랫폼에서 LLM 응답 로그나 대화 기록을 조회하는 API도 동일한 문제가 발생할 수 있습니다. 초기에 쿼리 수와 인덱스를 함께 설계하는 것이 훨씬 효율적임을 몸으로 배웠습니다.
🗄️ Fetch Join으로 N+1 해결 📈 복합 인덱스 2개 설계 ⚡ Redis 캐싱으로 DB 부하 제거 📐 쿼리 수 측정 습관화

게시글·좋아요·팔로우·실시간 채팅 기능을 가진 커뮤니티 웹앱을 Claude Code와 OpenAI Codex AI 에이전트를 기획·설계·구현·배포 전 과정에 투입하여 구현했습니다. AI가 ERD를 초안을 제안하면 제가 비즈니스 요구사항 관점에서 피드백하고, AI가 코드를 생성하면 저는 아키텍처 일관성과 보안을 검토하는 방식으로 협업했습니다. LLM이 오류를 감지·분석·수정하는 흐름을 직접 운영하면서, AI 플랫폼에서 셀프 힐링 엔진이 동작하는 방식을 실질적으로 이해하게 되었습니다.

콘텐츠 메타데이터 1,000건 이상을 매일 자동 수집하는 파이프라인을 구축할 때, JavaScript 동적 렌더링 페이지 크롤링 패턴, 네트워크 타임아웃 예외 처리, 재시도 로직을 AI와 함께 설계했습니다. 직접 모든 케이스를 예상하는 대신, AI가 생성한 패턴을 검토·수정·적용하는 방식으로 빠르게 안정적인 스크립트를 완성했습니다. Docker 컨테이너화와 Render Cron 스케줄 설정도 AI와 함께 최적 구성을 찾아냈고, 출시 이후 1년 이상 무중단 운영을 유지하고 있습니다.

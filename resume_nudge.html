01Required Stack

공고 필수 기술 스택 — Riverpod · GetX · BLoC

넛지헬스케어 공고에서 명시한 3가지 상태관리 라이브러리별 실무 경험입니다.

🥇
Riverpod
실서비스 적용 · 1년 10개월
매출 340억 OTT 서비스 오초이스에서 StateNotifierProvider로 STB 최대 3대의 연결·기기·콘텐츠 노출 상태를 단일 지점 관리. 상태 변화 시 홈·재생·설정 전체 화면 즉시 반영. Provider 간 의존성 설계, ConsumerWidget 활용 경험.
→ 헬로tv → 오초이스 마이그레이션 (LG헬로비전, 매출 1조) 프로젝트에 실서비스 적용
🥈
GetX
프로젝트 적용 경험
GetxController + Obx 반응형 UI 패턴, Get.to() 라우팅, GetBuilder 상태 관리 적용 경험. Riverpod 대비 경량·직관적인 라우팅과 의존성 주입이 필요한 상황에 적합함을 실무에서 판단해 선택적으로 사용.
→ 담아(Dama) 구독 관리 앱 등 개인·팀 프로젝트에 적용
🥉
BLoC
구조 이해 · 학습 적용
Event → BLoC → State의 단방향 데이터 흐름과 Stream 기반 처리 구조를 이해합니다. Riverpod의 StateNotifier와 동일한 불변 상태 원칙 위에 동작하며, WebSocket 실시간 스트림 처리 경험이 BLoC의 StreamSubscription 패턴과 직결됩니다.
→ Riverpod·WebSocket 실무 경험으로 BLoC 패턴 즉시 적용 가능
왜 3가지 상태관리를 다 아는 게 강점인가
각 라이브러리가 언제 더 적합한지 직접 판단할 수 있습니다. 기능 규모·팀 컨벤션·유지보수성을 고려해 상태관리 전략을 선택하는 능력은, 캐시워크처럼 복잡한 미션·리워드·걸음수 상태를 다루는 앱에서 특히 중요합니다.
02Problem Solving

문제 해결 — 캐시워크에 직결되는 경험

신입 포지션이지만 실서비스에서 마주한 문제들입니다. 캐시워크 앱 개발에 즉시 기여할 수 있습니다.

문제 Riverpod — 복잡한 상태 분리 설계

STB 3대의 상태(연결·기기·콘텐츠 노출)를 화면마다 중복 관리하다 상태 불일치가 발생했다

파트너사 앱 마이그레이션 초기에는 각 화면이 STB 상태를 개별로 관리했습니다. STB가 변경되면 일부 화면은 갱신되고 일부는 그대로 남아 상태가 어긋났습니다. 캐시워크에서 걸음수·미션 달성·리워드 잔액이 모든 화면에서 일관되게 보여야 하는 구조와 완전히 같은 문제입니다.

Riverpod 단일 상태 관리 — Before / After

STB 변경 이벤트
↓
홈 화면 상태 A
재생 화면 상태 B
설정 화면 상태 C
↓
화면마다 다른 값 ❌

STB 변경 이벤트
↓

↓ 구독 화면 자동 갱신
홈 · 재생 · 설정모두 동일한 값 ✅

해결StateNotifierProvider를 단일 진실 원천으로 정의하고, 모든 STB 이벤트를 이 Notifier에 집중시켰습니다. 각 화면은 watch만 하면 자동 갱신됩니다. 캐시워크의 걸음수 상태가 홈·랭킹·미션·리워드 화면에서 항상 동일한 값을 보여야 하는 구조에 동일하게 적용됩니다.

상태 불일치 완전 해소 StateNotifier 패턴 실서비스 검증 캐시워크 걸음수·미션·리워드 패턴 직접 전이
문제 Firebase Analytics · AdMob — 수익화 지표

사용자 이탈 구간을 특정하고, 광고 노출을 UX를 해치지 않는 위치에 배치하면서 수익과 리텐션을 동시에 잡아야 했다

OTT 앱에서 Firebase Analytics로 콘텐츠 선택 → 재생 시작 → 이탈 흐름을 추적했고, AdMob 광고를 사용자 흐름을 방해하지 않는 구간에 배치했습니다. 캐시워크의 광고 기반 리워드 모델, 미션 이탈 분석, 사용자 행동 추적과 완전히 같은 업무입니다.

Firebase Analytics 이벤트 설계 — 이탈 구간 특정
앱 진입
→
콘텐츠 탐색event 로깅
→
재생 시도이탈율 측정
→
재생 완료AdMob 노출
→
리텐션다음 세션

해결화면 전환 단위로 Firebase Analytics 이벤트를 설계해 정확한 이탈 구간을 측정했고, AdMob 배너/전면 광고를 재생 완료 후 자연스러운 전환 타이밍에 배치했습니다. 동시에 FCP 측정 기반으로 스켈레톤 UI·Lazy Loading·CDN을 적용해 FCP 39% 단축, LCP 10.8% 개선을 달성했습니다.

FCP −39% / LCP −10.8% Firebase Analytics 이벤트 설계 실무 AdMob UX 친화적 배치 경험
문제 실시간 동기화 — WebSocket · Stream 처리

다중 사용자의 동시 입력이 서버에 충돌 없이 반영되고, 결과가 모든 클라이언트에 즉시 전달되어야 했다

"숫자 대결" 실시간 게임에서 여러 사용자가 동시에 숫자를 바꾸면 충돌 없이 서버 최종값을 전체 클라이언트에 broadcast해야 했습니다. 캐시워크의 걸음수 달성 → 미션 완료 알림 즉시 반영, 실시간 랭킹 갱신 구조와 동일한 패턴이며, BLoC의 Stream 기반 처리와도 직결됩니다.

WebSocket 실시간 = BLoC Stream 처리 동일 패턴

해결서버 단일 상태 + WebSocket broadcast 구조로 다중 사용자 충돌을 해결했습니다. 기획·Flutter·Spring Boot·Render 배포 전 과정을 3일 내 단독 완성했습니다. Stream 기반 이벤트 처리 경험을 통해 BLoC의 단방향 데이터 흐름을 이해하고 즉시 적용할 수 있습니다.

WebSocket 실시간 동기화 구현 3일 내 기획~배포 완성 BLoC Stream 패턴 직접 전이 가능
문제 라이브러리 소스 레벨 디버깅

외부 플레이어 SDK 버그로 쇼츠 화면에서 오디오가 이중 출력됐고, 공식 패치를 기다릴 수 없었다

쇼츠 스크롤 시 특정 조건에서 플레이어가 두 번 초기화되어 오디오가 겹쳤습니다. pub.dev 이슈 추적만으로는 원인을 찾을 수 없어 라이브러리 소스를 직접 분석해야 했습니다.

단일 인스턴스 — 당일 배포 🔊

해결라이브러리를 fork하여 소스를 직접 분석했고 Android Activity 생명주기에서 중복 초기화 지점을 특정해 수정했습니다. 공식 수정 없이 당일 배포로 해결했습니다. 헬스케어 SDK, 만보기 API, 광고 SDK 이슈에도 동일한 접근 방식을 적용할 수 있습니다.

오디오 이중 출력 완전 해소 라이브러리 소스 레벨 분석 역량 당일 배포 해결

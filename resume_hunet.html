문제 WebView ↔ Native 연동 구조 설계

콘텐츠 선택·재생·구매 흐름이 WebView와 Native 앱 사이를 오가야 했다

콘텐츠 포스터가 많은 화면을 WebView(React)로 분리한 뒤, 사용자가 콘텐츠를 선택하거나 구매 버튼을 누르면 Native(Flutter)에서 재생·결제 흐름이 시작되어야 했습니다. 두 환경 사이에 데이터를 주고받는 구조가 없으면 화면이 동작을 멈추거나 상태가 어긋나는 문제가 발생했습니다.

해결 WebView에서 발생하는 이벤트를 JavaScript Bridge를 통해 Native로 전달하고, Native 처리 결과를 콜백으로 반환하는 양방향 구조를 설계했습니다. 콘텐츠 ID·사용자 세션 등 필요한 데이터를 직렬화하여 안전하게 전달하고, Native에서 화면 전환과 앱 상태를 관리하도록 역할을 분리했습니다.

앱 재배포 없이 운영 요구사항 즉시 반영 WebView · Native 역할 명확히 분리
문제 WebView 성능 저하

WebView로 전환한 후 초기 로딩이 느려져 사용자 이탈이 우려됐다

콘텐츠 포스터 화면을 WebView로 교체하자 초기 로딩 속도가 눈에 띄게 느려졌습니다. 원인을 파악하기 위해 FCP(First Contentful Paint)와 LCP(Largest Contentful Paint) 지표를 기준으로 병목 구간을 측정했습니다.

문제 이종 서비스 간 앱 마이그레이션

레거시 앱 2종의 구조를 분석하고 오초이스 앱으로 안정적으로 전환해야 했다

LG헬로비전의 "헬로tv" 앱과 딜라이브의 "딜라이브 모바일TV" 앱을 오초이스 앱으로 통합해야 했습니다. 두 앱은 각기 다른 화면 흐름, 재생 인터페이스, DRM 방식을 사용하고 있었고, 단순 기능 이전이 아닌 기존 사용자 경험을 유지하면서 전환해야 한다는 조건이 있었습니다.

해결 각 앱의 화면 진입 경로, 콘텐츠 재생 흐름, DRM 방식을 먼저 문서화하고, 오초이스 앱 구조와 어떻게 매핑되는지 정리했습니다. 사용자가 기존에 익숙했던 UI 패턴과 흐름을 최대한 유지하면서 내부 구조만 오초이스 방식으로 교체했습니다. WebView 하이브리드 구조를 적용하여 이후 운영 중 콘텐츠 화면을 앱 재배포 없이 수정할 수 있도록 했습니다.

2건 마이그레이션 완수 기존 사용자 경험 유지 Objective-C → Swift 전환에 직접 적용 가능한 경험
문제 DRM 호환성 문제

딜라이브(NCG DRM)와 오초이스(Multi DRM)의 방식이 달라 재생이 되지 않았다

딜라이브는 NCG DRM, 오초이스는 Multi DRM(Widevine·FairPlay 등)을 사용하고 있어 단순 플레이어로는 두 콘텐츠를 동시에 재생할 수 없었습니다. 별도의 DRM 솔루션을 도입하면 라이선스 비용과 외부 의존성이 생기는 문제도 있었습니다.

DRM 호환 구조

오초이스 콘텐츠 Multi DRM
↘↙

↓
두 DRM 모두 동일 UX로 재생

해결 별도 솔루션 도입 없이 플레이어 내부를 직접 커스터마이징하여 두 DRM 방식을 모두 처리할 수 있도록 분기 로직을 구성했습니다. 사용자에게는 동일한 재생 UI와 흐름을 제공하면서 내부에서만 DRM 처리 방식이 달라지도록 설계했습니다.

추가 라이선스 비용 없음 외부 솔루션 의존성 제거 사용자에게 단일 UX 제공
문제 셋톱박스 페어링 상태관리

하나의 앱에서 최대 3개 셋톱박스를 관리하며, 연결 상태에 따라 콘텐츠가 실시간으로 바뀌어야 했다

사용자가 연결된 셋톱박스를 변경하거나 연결·종료하면 앱 화면에 표시되는 콘텐츠가 즉시 바뀌어야 했습니다. 최대 3개까지 동시에 등록할 수 있어 상태 변화가 복잡했고, 화면 여러 곳에서 동일한 상태를 참조해야 했습니다.

최대 3개 셋톱박스 안정적 관리 콘텐츠 실시간 반영 상태 불일치 오류 제거
문제 Proxy · SSL 인프라 안정화

로드밸런싱 환경에서 Proxy 서버 IP가 바뀌어 통신이 끊어지는 문제가 발생했다

EC2 기반 Proxy 서버를 로드밸런서 뒤에 배치하는 과정에서 IP가 동적으로 변경되어 앱이 Proxy 주소를 제대로 찾지 못하는 문제가 발생했습니다. 또한 HTTP 구간이 남아 있어 보안 취약점이 존재했습니다.

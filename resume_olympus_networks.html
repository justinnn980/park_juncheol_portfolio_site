Section 02
실서비스에서 해결한 문제들
PROBLEM 01 · 웹/앱 동시 운영 구조
앱 배포 없이 콘텐츠 즉시 반영 — WebView-Native 하이브리드 구조
콘텐츠 변경 시 앱 재배포 필요 → 콘텐츠 탐색 화면 WebView 분리 → JS Bridge로 WebView-Native 데이터 동기화 → 스토어 심사 없이 즉시 반영
OTT 앱에서 콘텐츠·이벤트·배너가 자주 바뀌는데도 매번 앱 스토어 심사를 거쳐야 했던 문제를 React 웹 화면을 WebView로 분리하고 JavaScript Bridge로 양방향 데이터를 동기화하는 구조로 해결했습니다. 렌터카 요금 정책, 프로모션 배너처럼 자주 변경되는 화면을 가진 모빌리티 서비스에도 동일하게 적용 가능한 구조입니다.
변경 전
앱 코드 수정 → 빌드 → 스토어 심사(1~3일) → 배포 → 콘텐츠 반영
변경 후
React 웹 배포만으로 즉시 반영(앱 배포 불필요) · FCP −39% 달성
PROBLEM 02 · 백오피스 UI/UX 고도화
관리자 화면 UX 공백 — 낙관적 업데이트로 즉시 반응하는 백오피스
삭제 요청 후 서버 응답 대기 중 UI 멈춤 → TanStack Query 낙관적 업데이트 → 서버 실패 시 이전 상태 자동 복원 → 반응성 개선 + 데이터 안정성 확보
콘텐츠 관리 어드민에서 삭제 처리 시 서버 응답을 기다리는 동안 화면이 멈춰 보이는 문제를 onMutate에서 즉시 UI 반영, onError에서 이전 캐시로 자동 롤백하는 구조로 해결했습니다. 무인 배차·차량 컨디션 관리처럼 운영자가 실시간으로 상태를 다루는 백오피스 화면에도 동일하게 적용할 수 있습니다.
PROBLEM 03 · 서비스 통합 마이그레이션
이종 OTT 서비스 2건을 단일 앱으로 통합 — 안정적 마이그레이션 주도
서로 다른 파트너사 앱 2개 운영 부담 → 단일 Flutter 앱으로 마이그레이션 설계 → DRM·플레이어 구조 추상화로 통합 → 2개 파트너사 서비스 단일 앱 통합 완료
LG헬로비전(헬로tv), 딜라이브(모바일TV) 두 파트너사의 서로 다른 OTT 앱을 단일 서비스 앱으로 마이그레이션했습니다. DRM 방식(NCG/Multi DRM) 차이를 추상화한 플레이어 인터페이스로 통합 비용을 최소화했습니다. 신규 모빌리티 서비스를 기존 플랫폼에 통합·런칭하는 업무와 동일한 구조의 의사결정 경험입니다.
PROBLEM 04 · 앱스토어 배포 운영
스토어 배포·심사 전 과정 단독 수행
버전 관리·인증서·서명 비정형 업무 → iOS 인증서/프로비저닝 + Android Keystore 체계화 → TestFlight 배포 → 스토어 심사 대응 → 서비스 런칭·버전 관리 전 과정 단독 수행
OTT 앱 운영 기간 동안 iOS 인증서/프로비저닝, Android ProGuard·Keystore 서명, 스토어 심사 대응, 버전 관리를 단독으로 수행했습니다. 공고에서 요구하는 "서비스 런칭 및 앱스토어 배포 관리" 업무를 처음부터 끝까지 경험한 사례입니다.

Section 06
왜 올림포스네트웍스인가
REASON 01
모빌리티 도메인에 대한 기초 이해를 갖추고 있습니다
컴퓨터공학과 복수전공으로 모빌리티AI/SW융합을 이수했습니다. 렌터카·사고대차 시장의 데이터 기반 자동화라는 올림포스네트웍스의 방향성에 자연스럽게 관심을 갖고 있습니다.
REASON 02
웹과 앱을 함께 다루는 개발자가 필요한 조직입니다
사용자용 앱(Flutter), 사용자용 웹(React), 백오피스 UI/UX까지 모두 직접 경험했습니다. 한 사람이 웹/앱/백오피스를 넘나들며 빠르게 기능을 붙일 수 있는 역량이 신규 서비스를 빠르게 만들어가는 조직에 바로 기여할 수 있다고 생각합니다.

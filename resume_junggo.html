// StateNotifier = ViewModel, StateFlow = state class ContentNotifier extends StateNotifier<ContentState> { Future<void> loadContent(String userId) async { state = ContentState.loading(); try { state = ContentState.success(await _repo.getContent(userId)); } catch (e) { state = ContentState.error(e.toString()); } } }

Section 02
실서비스에서 맞닥뜨린 문제와 해결 과정
Native 전환 시에도 동일한 사고 방식으로 대응합니다.
PROBLEM 01
플레이어 라이브러리 소스 직접 분석 — 오디오 이중 출력 근본 해결 (→ ExoPlayer 이해)
스크롤 시 이전 영상 오디오 잔존 → API 문서로 해결 불가 판단 → 원본 라이브러리 소스 직접 분석 → 오디오 겹침 완전 해결
딜라이브 마이그레이션에서 쇼츠형 피드를 스크롤할 때 이전 영상의 오디오가 남아 겹쳐 출력되는 문제 발생. 단순 재생 메서드 재호출로는 플레이어 내부 생명주기를 제어하기 어렵다고 판단. 원본 플레이어 라이브러리 소스를 직접 열어 내부 초기화·재생·해제 흐름을 파악한 뒤, 영상 전환 시 생성 / 재생 / 종료 단계를 명시적으로 분리 호출하도록 수정해 완전 해결. Android Native에서 ExoPlayer·MediaPlayer 소스를 읽고 해결하는 사고 방식과 동일.
PROBLEM 02
Riverpod StateNotifier 도메인 분리 설계 — Android MVVM에 직접 이식 가능한 구조
3대 상태 혼재 · 렌더링 오류 → 연결/기기/콘텐츠 단위 분리 설계 → 상태 변화 즉시 반영 · 유지보수 명확
헬로tv 마이그레이션에서 셋톱박스 최대 3대 페어링 상태를 단일 Provider로 관리하면 상태가 혼재되어 잘못된 콘텐츠가 노출되는 문제 발생. 연결 상태 / 기기 노출 / 콘텐츠 접근 권한을 각각 별도 StateNotifierProvider로 분리해 각 단위의 상태가 독립적으로 변경·구독되도록 재설계. 이 설계 사고는 Android ViewModel을 Feature별로 분리하고 StateFlow로 단방향 데이터를 흘리는 구조와 동일.
Flutter (Riverpod)
StateNotifierProvider를 연결/기기/콘텐츠 단위로 분리 → 각 상태 독립 관리
Android (MVVM)
ViewModel을 Feature별로 분리 → StateFlow로 단방향 데이터 흐름 관리
PROBLEM 03
NCG DRM → Multi DRM 전환 — 플레이어 생명주기 이해 기반 단일 인터페이스 통합
서비스마다 다른 DRM 방식 → DRM 추상화 레이어 설계 → 단일 플레이어 인터페이스 통합 → 서비스 무관 동일 사용자 경험
딜라이브 앱(NCG DRM)과 헬로tv 앱(Multi DRM: Widevine/FairPlay) DRM 방식이 달라 재생 흐름이 서비스마다 달라지는 문제. 서비스별로 별도 플레이어를 만들면 유지보수 비용이 두 배가 된다고 판단. 기존 플레이어를 커스터마이징해 DRM 방식을 내부에서 추상화하고 사용자에게는 동일한 UI·요청 흐름으로 콘텐츠를 재생할 수 있도록 단일 인터페이스로 통합. ExoPlayer에서 DRM 세션 관리 구조와 직접 연결되는 경험.
PROBLEM 04
WebSocket 실시간 서버 상태 동기화 — Kotlin 서버 + Flutter 클라이언트 3일 단독
동시 입력 충돌 · 상태 불일치 가능성 → 서버 단일 상태 원천 설계 → Optimistic Update + 서버 확정 → 충돌 없는 실시간 동기화
실시간 숫자 대결 게임에서 여러 클라이언트가 동시에 입력하면 상태 불일치가 발생할 수 있는 문제. 서버를 단일 상태 원천(Single Source of Truth)으로 설계해 모든 상태 변경을 서버 경유로 처리. 클라이언트는 Optimistic Update로 즉각 UI 반응을 주고, 서버 확정값으로 동기화. Kotlin Spring Boot WebSocket 서버와 Flutter 클라이언트를 기획부터 배포까지 3일 내 단독 구현.
PROBLEM 05
앱 배포 주기 의존 탈피 — WebView JS Bridge 양방향 구조 설계, FCP 39% 단축
화면 변경마다 앱 재배포 필요 → React + WebView 분리 → FCP/LCP 기반 최적화 → 즉시 반영 운영 + FCP −39%
콘텐츠 구성·배너·화면 변경마다 앱 재배포가 필요해 운영 비용이 높고, WebView 전환 후 초기 로딩이 느려지는 부작용도 발생. React 기반 웹으로 탐색 화면을 분리하고 Flutter ↔ WebView JavaScript Bridge 양방향 통신 구조를 설계해 앱 배포 없이 즉시 반영 가능한 운영 구조 구현. FCP/LCP 지표로 병목을 찾아 Lazy Loading·스켈레톤 UI·CDN을 적용해 FCP 39% 단축, LCP 10.8% 개선.
PROBLEM 06
Flutter 2 → 3 메이저 버전업 — 전체 의존성 재검토 후 안정적 완료
Flutter 3 업그레이드 필요 → DRM·플레이어·상태관리 전수 검토 → 비호환 라이브러리 교체 · Null Safety → 서비스 중단 없이 업그레이드 완료
딜라이브 마이그레이션 중 Flutter 2 기반 코드베이스의 Flutter 3 업그레이드 필요. 단순 버전 태그 변경으로 끝나지 않고 DRM 라이브러리, 미디어 플레이어, Riverpod, 기타 의존 라이브러리의 Flutter 3 호환 여부를 전수 검토. 비호환 라이브러리 교체 버전 확인 및 Null Safety 마이그레이션이 필요한 코드를 직접 수정해 기능 이상 없이 안정적으로 업그레이드 완료. 중고나라 Flutter to Native 리빌딩에서 코드베이스 전환 시 동일한 접근 방식으로 안정성을 확보할 수 있습니다.
